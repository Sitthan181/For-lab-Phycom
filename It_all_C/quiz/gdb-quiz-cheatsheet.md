# GDB Cheat Sheet — เตรียมสอบ Quiz

เอกสารนี้เน้น **คำสั่งและหลักการ** เป็นหลัก ไม่ผูกกับรูปแบบโจทย์ใดโจทย์หนึ่ง เพราะยังไม่รู้ว่า quiz จริงจะออกมาแบบไหน — แต่ไม่ว่าโจทย์จะถามอะไร ขั้นตอนการใช้ gdb หาคำตอบจะไม่ต่างจากนี้มากนัก

---

## 1. ตารางคำสั่ง GDB ทั้งหมด (อ้างอิงหลัก)

| คำสั่ง | ตัวย่อ | รูปแบบการใช้งาน | คำอธิบาย |
|---|:---:|---|---|
| **list** | `l` | `list` / `l <line>` / `l <func>` / `l -` | แสดงโค้ด 10 บรรทัด รอบจุดปัจจุบัน/บรรทัด/ฟังก์ชันที่ระบุ; `list -` = ถอยไปดู 10 บรรทัดก่อนหน้าที่แสดงล่าสุด |
| **break** | `b` | `break <line>` / `b <func>` / `b <line> if <cond>` | ตั้งจุดหยุด (breakpoint) ที่บรรทัด/ฟังก์ชัน หรือแบบมีเงื่อนไข |
| **run** | `r` | `run` | เริ่มรันโปรแกรมตั้งแต่ต้นจนถึง breakpoint แรก หรือจบโปรแกรม |
| **print** | `p` | `print <expr>` | แสดงค่าตัวแปรหรือนิพจน์ ณ ขณะนั้น (ใส่นิพจน์ซับซ้อนได้ เช่น `print a+b`) |
| **info locals** | - | `info locals` | แสดงค่าตัวแปร local **ทั้งหมด**ในสโคปปัจจุบันทีเดียว (ไม่ต้อง print ทีละตัว) |
| **info breakpoints** | `i b` | `info breakpoints` | แสดงรายการ breakpoint/watchpoint ทั้งหมดที่ตั้งไว้ |
| **info args** | - | `info args` | แสดงค่า parameter ที่ฟังก์ชันปัจจุบันได้รับ |
| **delete** | `d` | `delete` / `delete <num>` | ลบ breakpoint/watchpoint; ไม่ใส่เลข = ลบทั้งหมด |
| **disable / enable** | - | `disable <num>` / `enable <num>` | ปิด/เปิดใช้งาน breakpoint โดยไม่ต้องลบทิ้ง |
| **next** | `n` | `next` | รันบรรทัดถัดไป **ไม่เข้าไปในฟังก์ชัน** ที่ถูกเรียก |
| **step** | `s` | `step` | รันบรรทัดถัดไป **เข้าไปในฟังก์ชัน** ที่ถูกเรียก |
| **continue** | `c` | `continue` | รันต่อเนื่องจนกว่าจะเจอ breakpoint ถัดไป หรือจบโปรแกรม |
| **finish** | `fin` | `finish` | รันจนจบฟังก์ชันปัจจุบัน หยุดที่บรรทัดถัดจากจุดเรียก |
| **watch** | - | `watch <expr>` | หยุดโปรแกรมทันทีที่ค่าของตัวแปร/นิพจน์นั้น**เปลี่ยนแปลง** (โชว์ Old value / New value) |
| **info watchpoints** | - | `info watchpoints` | แสดงรายการ watchpoint ทั้งหมด |
| **quit** | `q` | `quit` | ออกจาก GDB |

**หลักการจำตัวย่อ**: ส่วนใหญ่คือ**ตัวอักษรแรกของคำสั่ง** (`b`reak, `p`rint, `n`ext, `s`tep, `c`ontinue, `r`un, `q`uit, `l`ist) ยกเว้น `finish`→`fin` และ `info`→`i` (คำสั่งย่อยของ `info` ก็พิมพ์แบบย่อได้ เช่น `i b` = `info breakpoints`, `i loc` = `info locals` — gdb รองรับ prefix แบบไม่กำกวม แต่ถ้าไม่มั่นใจให้พิมพ์เต็มปลอดภัยกว่า)

## 2. ขั้นตอนมาตรฐานก่อนเริ่ม Debug (ทำทุกครั้งที่แก้โค้ด)

```bash
# 1. เขียน/แก้โค้ดด้วย vim
vim main.c

# 2. Compile พร้อม flag -g (ขาดไม่ได้ ไม่งั้น debug ไม่ได้)
gcc -g main.c -o main

# 3. เปิด gdb
gdb main
```

### เข้าโหมด TUI (แบ่งจอเห็นโค้ด — แนะนำให้ใช้ตลอด)

```
Ctrl + x   แล้วตามด้วย   a
```

กดซ้ำได้เพื่อสลับเข้า/ออก TUI ตลอดเวลา (ใช้ได้เสถียรกว่าเปิดด้วย `gdb --tui` ตรงๆ บน Cygwin/mintty)

## 3. หลักการทั่วไปในการหาคำตอบ (ปรับใช้ได้กับโจทย์ทุกแบบ)

ไม่ว่าโจทย์จะถามอะไร (ค่าตัวแปรก่อน/หลัง, ค่าใน loop, ค่าที่ return, ที่อยู่ memory ฯลฯ) แนวทางหลักคือ:

1. **หาว่าอยากรู้ค่า ณ "จุดไหน" ของโปรแกรม** → นั่นคือบรรทัดที่ต้องตั้ง `break`
2. **`break <line>` แล้ว `run`** ให้โปรแกรมมาหยุดตรงจุดนั้นก่อน
3. **`print <var>` หรือ `info locals`** เพื่อดูค่า ณ จุดที่หยุด
4. ถ้าต้องดูค่า "หลัง" บรรทัดนั้นทำงานเสร็จ → ใช้ `next` (ถ้าไม่อยากเข้าไปดูข้างในฟังก์ชัน) แล้ว `print`/`info locals` ซ้ำ
5. ถ้ามีหลายจุดที่ต้องดูในโค้ดเดียวกัน (เช่น หลายบรรทัดในตารางเดียวกันของ 1 ข้อ) → ตั้ง `break` ไว้ **ทุกจุดตั้งแต่แรก** แล้วใช้ `continue` ไล่จากจุดหนึ่งไปอีกจุดหนึ่ง (ไม่ต้อง `run` ใหม่ทุกรอบ)
6. ถ้าอยากดูว่าตัวแปรไหนถูกเปลี่ยนค่าตรงไหนบ้างโดยไม่ต้องไล่ `next` ทีละบรรทัดเอง → ใช้ `watch <var>` แล้ว `continue` ไปเรื่อยๆ gdb จะหยุดเองทุกครั้งที่ค่านั้นเปลี่ยน พร้อมโชว์ Old/New value ให้เลย

### กรณีต้องรันโปรแกรมใหม่ตั้งแต่ต้น (ไม่ต้องออกจาก gdb)

```gdb
(gdb) run
```
ถ้าเคยรันไปแล้ว gdb จะถาม:
```
The program being debugged has been started already.
Start it from the beginning? (y or n) y
```
พิมพ์ `y` แล้ว Enter — breakpoint เดิมที่ตั้งไว้จะยังอยู่ครบ ไม่ต้องตั้งใหม่

## 4. จุดที่ต้องระวังเรื่อง Scope ของตัวแปร (สำคัญกับ `info locals` และ `print`)

- `info locals` และ `print` จะอ้างอิงค่าตัวแปร**ตาม scope ปัจจุบัน ณ บรรทัดที่โปรแกรมหยุดอยู่เท่านั้น**
- ถ้าโค้ดมีการเปิด `{ }` ซ้อน (nested block) แล้วมีการ**ประกาศตัวแปรชื่อซ้ำกับตัวแปรนอก** (variable shadowing) — ตัวแปรในสุดจะถูกใช้ก่อนเสมอ แม้ตัวแปรนั้นจะยังไม่ถูก initialize ก็ตาม (จะได้ค่าขยะ ไม่ใช่ error)
- **ถ้าโจทย์บอกว่า statement ในตารางเดียวกันอยู่ในโค้ดเดียวกัน** ให้เขียนทุก statement เรียงต่อกันตามลำดับในโค้ดเดียว (ไม่เปิด scope ใหม่ ไม่ตั้งชื่อตัวแปรซ้ำ) เพื่อให้ค่าตัวแปรสะสมต่อเนื่องกันถูกต้องตามที่ควรจะเป็น

## 5. ความรู้ภาษา C ที่ควรทบทวนคู่กัน (เพราะ quiz แนวนี้มักเทสเรื่องนี้)

### Pre-increment vs Post-increment
| นิพจน์ | พฤติกรรม |
|---|---|
| `a++` (post) | ใช้ **ค่าเดิม** ของ `a` ในนิพจน์ก่อน แล้วค่อยเพิ่มค่า `a` ทีหลัง |
| `++a` (pre) | เพิ่มค่า `a` **ก่อน** แล้วค่อยใช้ค่าใหม่ในนิพจน์ |

### Operator ที่ควรรู้ทางคณิตศาสตร์
- `/` ระหว่าง `int` กับ `int` = **หารจำนวนเต็ม (ตัดเศษทิ้ง ไม่ปัด)**
- `%` = หาเศษจากการหารจำนวนเต็ม
- ลำดับการคำนวณ (`*`, `/`, `%` มาก่อน `+`, `-`) และคำนวณจากซ้ายไปขวาเมื่อลำดับเท่ากัน

### Undefined Behavior ที่พบบ่อยในนิพจน์แบบ `a = a++ + ++a;`
นิพจน์ที่แก้ไขตัวแปรเดียวกันมากกว่า 1 ครั้งโดยไม่มี sequence point คั่น (เช่น `,` หรือ `;`) เป็น **Undefined Behavior** ตามมาตรฐาน C — ผลลัพธ์ที่ได้**ขึ้นอยู่กับ compiler แต่ละตัว** ไม่มีคำตอบที่ "ถูกต้องตามทฤษฎี" ตายตัว **นี่คือเหตุผลหลักที่โจทย์บังคับให้ใช้ gdb หาคำตอบจริงจากเครื่องที่ใช้สอบ แทนที่จะคำนวณด้วยมือ** เพราะคำตอบที่คำนวณเองอาจไม่ตรงกับที่ compiler จริงให้ผลออกมา

## 6. Troubleshooting ที่เจอบ่อย

| ปัญหา | สาเหตุ / วิธีแก้ |
|---|---|
| ค้างที่ `--Type <RET> for more, q to quit--` | ข้อความยาวเกิน 1 หน้าจอ (pager) → กด `q` เพื่อข้ามออกมา |
| `gdb: command not found` | gdb ยังไม่ได้ติดตั้ง → เปิด Cygwin Setup ติดตั้ง package `gdb`, `libgdbm-devel`, `libgdbm4`, `libgdbm6`, `libgdbm_compat4` |
| ติดตั้ง Cygwin แล้วขึ้น "In-use file detected" | มี terminal เปิดค้างอยู่ใช้ไฟล์ระบบอยู่ → กด **Stop Processes** แล้วเปิด terminal ใหม่ |
| `gdb --tui main` เปิดแล้วปิดตัวเองทันที | ปัญหา mintty (Cygwin terminal) กับ TUI → เปิด `gdb main` แบบปกติก่อน แล้วกด `Ctrl+x` ตามด้วย `a` แทน |
| `gdb -- version` (มีช่องว่าง) พังคนละแบบกับที่คิด | เพราะกลายเป็นคำสั่งเปิดไฟล์ชื่อ "version" มา debug ไม่ใช่ flag `--version` → ต้องพิมพ์ `gdb --version` ติดกัน |
| ค่าที่ print ได้เป็นค่าประหลาด/ค่าขยะ | มักเกิดจาก scope ซ้อนกันและตัวแปรถูก shadow (ดูหัวข้อ 4) หรือ breakpoint หยุดผิดจุด (หยุดก่อนตัวแปรถูก initialize) |
| แก้โค้ดแล้วค่าไม่เปลี่ยนตอน debug | ลืม compile ใหม่ก่อนเปิด gdb → ต้อง `gcc -g main.c -o main` ทุกครั้งที่แก้ไฟล์ |

## 7. Checklist ก่อนเข้าห้องสอบ

- [ ] จำ flag `-g` ตอน compile ได้ (`gcc -g main.c -o main`)
- [ ] เข้า TUI ได้คล่อง (`Ctrl+x` แล้ว `a`)
- [ ] ตั้ง breakpoint หลายจุดพร้อมกันได้ (`break <line>` ซ้ำหลายครั้งก่อน `run`)
- [ ] ใช้ `next` vs `step` ถูกจังหวะ (ปกติใช้ `next` พอ ยกเว้นโจทย์มีเรียกฟังก์ชันแล้วอยากดูข้างใน)
- [ ] ใช้ `continue` ไล่ระหว่าง breakpoint ได้ โดยไม่ต้อง `run` ใหม่ทุกรอบ
- [ ] ใช้ `info locals` แทน `print` ทีละตัวได้ (ประหยัดเวลา)
- [ ] เข้าใจเรื่อง pre/post increment และรู้ว่านิพจน์ซับซ้อนบางแบบเป็น Undefined Behavior (เหตุผลที่ต้องพึ่ง gdb แทนคำนวณมือ)
- [ ] รู้วิธีแก้ปัญหาเบื้องต้นถ้า gdb ค้างหรือ terminal มีปัญหา (ดูหัวข้อ 6)
